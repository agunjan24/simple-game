  Start niche, build a loyal Bollywood audience, then rebrand to "CineFrames"
  when you add enough themes to justify it. Risk: rebranding means losing app
  store reviews, search ranking, and brand recognition.

  Option C: "Bollywood Frames" as a sub-brand
  Parent brand "CineFrames" with themed editions: "CineFrames: Bollywood,"
  "CineFrames: Hollywood." This works in marketing but feels over-engineered for
   an app store listing.

  My recommendation: Go with a generic name like CineFrames from the start. It's
   easier to broaden than to rebrand. Bollywood fans will still find it — the
  Bollywood theme inside the app can be rich and immersive. The app store
  listing can mention "Bollywood, Hollywood, and more" in the description.
  Renaming later is painful (new bundle ID, lost reviews, confused users).

  2. Expanding beyond movies

  The core mechanic — "look at a visual cue, guess the thing" — is genuinely
  versatile. The question is same app or separate apps.

  Same app, multiple categories

  Pros:
  - Single codebase, single deployment
  - Cross-pollination: a movie fan discovers the history mode
  - Larger user base for a single app store listing (more reviews, higher
  ranking)
  - Shared infrastructure (scoring, teams, timer, progressive reveal all reuse)

  Cons:
  - App identity gets diluted — "what is this app about?"
  - App store optimization is harder (you can't rank #1 for both "movie quiz"
  and "history quiz")
  - Feature creep risk: each category may need slightly different mechanics
  (history might need date-based scoring, logic puzzles might not need a timer)
  - Larger app size as content grows

  Separate apps, shared framework

  Pros:
  - Each app has a clear identity and targeted app store listing
  - Can tailor mechanics per domain without compromise
  - If one app fails, it doesn't drag down the others
  - Smaller download size per app

  Cons:
  - Multiple codebases to maintain (even with a shared library, there's
  overhead)
  - Multiple app store listings to manage
  - Users who'd enjoy both have to discover and install separately

  My take

  For now, design the data model generically but ship as one app. Your current
  architecture already supports this naturally — a "theme" is just a config with
   colors, data file, image folder, and team names. A history theme or logic
  theme is structurally identical: different images, different answers,
  different color palette.

  The key abstraction to get right early is terminology. Right now the code says
   "movie," "movie_name," "hint." If you rename these internally to something
  generic — "item," "answer," "clue" — adding non-movie categories later is just
   a new theme entry with no code changes.

  If the app grows large enough that the identity problem becomes real (users
  confused about what the app is), that's a good problem to have, and you can
  split at that point. Splitting one app into two is much easier than merging
  two apps into one.
